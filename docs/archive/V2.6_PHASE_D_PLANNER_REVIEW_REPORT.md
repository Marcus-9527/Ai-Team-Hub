# V2.6 Phase D — Planner Review Layer

**Date:** 2026-07-07  
**Status:** ✅ Complete  
**Branch:** (main)

---

## Overview

Added a **Review Gate** between TaskPlan generation and TaskStep application. This enforces a human-in-the-loop check before any Planner-generated plan can be converted into executable steps.

### Flow Before

```
Planner → TaskPlan(ACTIVE) → convert_plan_to_steps → TaskStep
```

### Flow After

```
Planner → TaskPlan(ACTIVE) → request_review → approve_review → convert_plan_to_steps → TaskStep
                                   ↓ (if rejected)
                              Plan blocked, cannot apply
```

## What Was Added

### 1. `TaskPlanReviewModel` (`backend/models.py`)

| Field       | Type     | Description                          |
|-------------|----------|--------------------------------------|
| `id`        | String   | PK, UUID                             |
| `plan_id`   | String   | FK → task_plans.id (unique, CASCADE) |
| `status`    | String   | PENDING / APPROVED / REJECTED        |
| `reviewer`  | String   | Who performed the review             |
| `comment`   | Text     | Review note / rejection reason       |
| `created_at`| DateTime | Auto timestamp                       |

### 2. `TaskPlanReviewService` (`backend/services/task/task_plan_review.py`)

New service with four public methods:

| Method             | Description                                       |
|--------------------|---------------------------------------------------|
| `request_review()` | Create a PENDING review for an ACTIVE plan         |
| `approve_review()` | Approve a PENDING review → status APPROVED         |
| `reject_review()`  | Reject a PENDING review → status REJECTED          |
| `check_review_status()` | Return current review status (or None)        |
| `require_approval()`    | **Review Gate** — raise unless status is APPROVED |

**Error hierarchy:**
- `ReviewError` (base)
- `ReviewNotFoundError` — no review exists
- `ReviewNotPendingError` — trying to approve/reject a non-PENDING review
- `ReviewGateBlockedError` — plan not yet approved

### 3. Review Gate (`backend/services/task/task_plan_service.py`)

Modified `convert_plan_to_steps()` to call `self.review.require_approval()` **before** any step conversion. The gate checks:

1. **No review exists** → `ReviewGateBlockedError` ("has not been reviewed")
2. **Review is PENDING** → `ReviewGateBlockedError` ("review is PENDING")
3. **Review is REJECTED** → `ReviewGateBlockedError` ("review was REJECTED")
4. **Review is APPROVED** → pass through to conversion logic

The gate sits **before** policy checks (empty-plan, risk-level, cost), so the review is the first guard.

### 4. API Endpoints (`backend/routes/tasks.py`)

| Method | Path                              | Description                      |
|--------|-----------------------------------|----------------------------------|
| POST   | `/api/tasks/{id}/plan/review`     | Request review for ACTIVE plan   |
| POST   | `/api/tasks/{id}/plan/approve`    | Approve a PENDING review         |
| POST   | `/api/tasks/{id}/plan/reject`     | Reject a PENDING review          |

All three endpoints:
- Return `PlanReviewResponse` (id, plan_id, status, reviewer, comment, created_at)
- Return 404 if task/plan not found
- Return 400 on invalid state transitions

The existing `POST /api/tasks/{id}/plan/apply` now returns **403** if the review gate is not passed (in addition to the existing 403 for policy blocks).

### 5. Database Registration (`backend/database.py`)

Added `TaskPlanReviewModel` import to `init_db()` so the `task_plan_reviews` table is auto-created.

## Quick Start

```bash
# 1. Create a plan (assumes task exists)
POST /api/tasks/{task_id}/plan

# 2. Request review
POST /api/tasks/{task_id}/plan/review
{"reviewer": "admin", "comment": "Check this plan"}

# 3. Approve
POST /api/tasks/{task_id}/plan/approve
{"reviewer": "admin", "comment": "Approved"}

# 4. Apply (now passes review gate)
POST /api/tasks/{task_id}/plan/apply

# Alternative: reject
POST /api/tasks/{task_id}/plan/reject
{"reviewer": "admin", "comment": "Needs more detail"}
```

---

## Caveats

- The `request_review` is **idempotent** — calling it again returns the existing review.
- A **rejected** plan cannot be applied until a new review is created (e.g. by re-generating the plan or manually updating the review status in DB).
- The `task_plan_reviews` table is created automatically on next server restart due to `Base.metadata.create_all`.
