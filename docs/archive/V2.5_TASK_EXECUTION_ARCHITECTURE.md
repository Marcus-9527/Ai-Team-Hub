# AI Team Hub v2.5 Task Execution Layer Architecture

> **阶段**: 架构设计  
> **日期**: 2026-07-07  
> **目标**: 在 v2.4 MAEOS 之上设计 Task Execution Layer，使 AI Team Hub 从"AI 协作聊天平台"升级为"AI Team Execution Platform"  
> **原则**:
> - 不修改现有核心代码（maeos、runtime、kernel）
> - 不创建第二套 Agent Runtime
> - 所有 Agent 执行必须经过 MAEOS Runtime
> - 保持 v2.4 稳定

---

## 1. 当前架构分析

### 1.1 当前执行链路

```text
User Request
     │
     ▼
Channel API (routes/channels.py)
  ── SSE stream: team_collaboration.py (multi‑teammate sequential chain)
  ── REST: channel CRUD (legacy)
     │
     ▼
MAEOS (services/maeos.py)
  ── PriorityTaskQueue (priority‑sorted FIFO)
  ── Worker Pool (FSMWorker × N)
  ── ExecutionMemory (in‑memory, max 1000 entries)
     │
     ├─► Scheduler Loop
     │     ① pop() highest‑priority task
     │     ② assign to available FSMWorker
     │     ③ asyncio.create_task(worker.execute(task))
     │
     ├─► FSMWorker.execute()
     │     ① task.status = RUNNING
     │     ② run_pipeline(planner → executor → reviewer)
     │     ③ task.status = COMPLETED / FAILED
     │     ④ save to ExecutionMemory
     │     ⑤ signal completion event
     │
     ▼
Runtime (services/runtime/)
  ├── Scheduler          — ExecUnit queue, dispatch, concurrency
  ├── RetryPolicy        — failure classification + backoff
  ├── TraceLogger        — structured JSON trace events
  ├── ContextIsolation   — per‑teammate context isolation
  └── FlowControlEnforcer — hard rule enforcement
```

```text
  ── ExecutionMemory
```

**核心洞察**：Task Layer 是 MAEOS 之上的编排层，**不是替代 Runtime**。每个 TaskStep 最终调用 MAEOS.submit() 触发在现有 Worker Pool 上执行。

## 2. Task Layer 架构设计

### 2.1 模块结构

```
backend/services/task/
├── __init__.py
├── task_manager.py      — Task CRUD + lifecycle orchestration
├── task_planner.py      — User goal → TaskStep decomposition
├── task_executor.py     — Step dispatch to MAEOS + result collection
├── task_state.py        — State machine + persistence
├── task_scheduler.py    — Step sequencing + dependency resolution
├── task_review.py       — Step output validation + human approval
└── task_memory.py       — Task‑level memory context builder
```

### 2.2 各模块职责

| 模块 | 职责 | 输入 | 输出 |
|---|---|---|---|
| **task_manager.py** | Task CRUD、生命周期、暂停/恢复/取消 | User goal + workspace_id + channel_id | Task 对象、事件通知 |
| **task_planner.py** | 将用户目标分解为有序的 TaskStep | User goal + intent + context | list[TaskStep] |
| **task_executor.py** | 遍历 TaskStep，逐个提交到 MAEOS，收集结果 | TaskStep + MAEOS instance | step 结果、执行记录 |
| **task_state.py** | Task/Step 状态机 + DB 持久化 | task_id, step_id | 状态变更事件 |
| **task_scheduler.py** | Step 依赖解析 + 并发/串行调度策略 | list[TaskStep] + dependencies | 可执行 step 队列 |
| **task_review.py** | Step 输出校验 + 质量门禁 + 人工确认请求 | step output + review rules | {pass, fail, need_confirm} |
| **task_memory.py** | 构建 Task 级别的 Memory Context，注入到每个 Step | workspace_memory + channel_memory + global_memory | context_blob |

### 置要件

```
禁止：
  ❌ 创建第二套 Agent Runtime
  ❌ 绕过 MAEOS 直接调用 LLM
  ❌ 修改 v2.4 现有核心代码（maeos.py、runtime/*、kernel/*）

必须：
  ✅ 所有 Agent 执行必须经过 MAEOS Runtime
  ✅ Task Layer 是 MAEOS 的客户端/编排层
  ✅ 复用现有的 RetryPolicy、TraceLogger、ContextIsolation
```

### 2.3 Task Layer 与 MAEOS 的集成

```text
Task Layer                              MAEOS
──────────                              ─────
task_executor.py ── submit(step) ──►   PriorityTaskQueue
                                           │
    ◄── wait(task_id) ────────────────  FSMWorker.execute()
                                           │
    ◄── result / error ───────────────  ExecutionMemory.save()
```

TaskExecutor 本身**不创建任何 Runtime 实例**。它调用 `MAEOS.submit()` 将每个 step 以 task 形式提交到现有队列。

每步执行流程：

```
for each step in steps:
    task_id = maeos.submit(description=step.objective, ...)
    task = await maeos.wait(task_id)
    if task.status == COMPLETED:
        step.output = task.result
        task_state.update(step, DONE)
    else:
        task_state.update(step, FAILED)
        review.decide(step, task.error)
```

### 3.2 与现有表的关系


### 3.3 状态机

#### Task 状态

```text
PENDING ──► PLANNING ──► EXECUTING ──► REVIEWING ──► COMPLETED
                  │            │            │
                  ▼            ▼            ▼
               FAILED ◄── retry ──── retry ◄── FAILED (review not pass)
                  │
                  ▼
              CANCELLED              PAUSED ◄── any state
                                         │
                                         ▼
                                    (resume → back to previous state)
```

#### TaskStep 状态

```text
PENDING ──► SCHEDULED ──► RUNNING ──► COMPLETED
                               │
                               ▼
                            FAILED ──► retry? ──► SCHEDULED (if retry)
                               │
                               ▼
                            SKIPPED (downgrade)
```

## 5. 执行流程

### 5.1 Chat 模式（保持不变）

```text
User ──► Channel (SSE stream)
           │
           ▼
    team_collaboration.py
      ── 角色链式调用（engineer → analyst → ...）
      ── 直接通过 ai_service 调用 LLM
      ── SSE events: teammate_message, teammate_end
```

### 5.2 Task 模式（新增）

```text
User Goal
     │
     ▼
Intent Classification ◄── question/code/analysis/planning?
  │
  ├─ "chat" ──► existing Chat flow
  │
  └─ "task" ──► POST /api/tasks
                     │
                     ▼
              task_manager.create_task()
                     │
                     ▼
              task_planner.plan(user_goal, context)
                     │
                     │  ┌──────────────────┐
                     │  │ Step 1: Research   │ (teammate_a)
                     │  │ Step 2: Implement  │ (teammate_b)
                     │  │ Step 3: Review     │ (teammate_j)
                     │  └──────────────────┘
                     │
                     ▼
              task_state.save(steps)
              task_state.set_status(PLANNING → EXECUTING)
                     │
                     ▼
              task_scheduler.resolve_order(steps)
                     │
                     ▼
              task_executor.execute_loop():
                for each step in ordered_steps:
                     │
                     ▼
                  task_memory.build_context(step, workspace_memory)
                     │
                     ▼
                  maeos.submit(step.objective, context)
                     │
                     ▼
                  await maeos.wait(task_id)
                     │
                     ├─ success ──► step.output = result
                     │                  │
                     │                  ▼
                     │           task_review.validate(step)
                     │                  │
                     │           ├─ pass ──► next step
                     │           └─ fail ──► retry / human_approval
                     │                        │
                     │                   need_confirm?
                     │                        │
                     │                        ▼
                     │                 SSE: approval_required
                     │                        │
                     │                  wait for user action
                     │
                     └─ failure ──► task_review.decide()
                                       ├─ retry ──► maeos.submit again
                                       ├─ downgrade ──► skip step
                                       └─ abort ──► task FAILED
                     │
                     ▼
              task_state.set_status(COMPLETED / FAILED)
                     │
                     ▼
              SSE: task_completed / task_failed
```

### 5.3 Intent Classification 策略

新增 `backend/services/intent_classifier.py`（轻量模块），在渠道层（Channel route / SSE handler）拦截：

```python
async def classify_intent(message: str, context: dict) -> str:
    """
    判断用户输入是 Chat 还是 Task。

    规则：
    1. 如果消息包含明确的任务关键词 → "task"
```

## 6. 与 MAEOS 集成方案

### 6.1 集成点

| 集成点 | 方向 | 说明 |
|---|---|---|
| `MAEOS.submit()` | Task → MAEOS | TaskExecutor 调用，传递 step objective |
| `MAEOS.wait()` | Task → MAEOS | 等待单个 step 执行完成 |
| `MAEOS.get_status()` | Task → MAEOS | 轮询 step 执行状态 |
| `MAEOS.memory` | Task → MAEOS | 读 ExecutionMemory 获取 trace/diversity 报告 |
| `Runtime.RetryPolicy` | Task → Runtime | TaskExecutor 复用（非直接，通过 MAEOS 内部触发） |
| `Runtime.TraceLogger` | Task → Runtime | 通过 maeos_task_id 关联 trace |
| `SharedContext` | Task → Collaboration | 可选：Task 执行过程中写入 event-sourced context |

### 6.2 无侵入要求

```
MAEOS 侧：
  ✅ 无修改。Task Layer 是 MAEOS 的纯客户端。
  ✅ 所有 MAEOS API 保持原样。

Task Layer 侧：
  ✅ 通过 MAEOS 实例引用调用 submit/wait。
  ✅ 不继承/覆盖 MAEOS 内部类。
  ✅ 不注入 monkey-patch。
```

### 6.3 选择不侵入的理由

1. **隔离风险**：Task Layer 可能在迭代中出 bug，与现有聊天通道隔离，不会影响正在进行的聊天
2. **渐进部署**：可以先上线 Task API，逐步开放
3. **清晰职责**：MAEOS = 单次执行引擎，Task = 多步编排。分开后各自保持内聚

## 7. Memory 集成设计

### 7.1 Memory 层级

```text
Global Memory                    ── 全局配置/常识（system prompts）
     │
     ▼
Workspace Memory                 ── workspace_memory.py（基于 SharedContext）
  ── 决策历史
  ── 对话历史
  ── 推理追踪
  ── 修订历史
     │
     ▼
Channel Memory                   ── 当前 channel 的消息历史（Message 表）
     │
     ▼
Task Context (new)               ── 一次 Task 执行的上下文
  ── task.description + intent
  ── 前序 steps 的输出
  ── workspace_memory 中相关条目
  ── channel 消息摘要
     │
     ▶ 注入到每个 TaskStep 的 input_context
```

### 7.2 Task Memory 构建流程

```python
# task_memory.py
async def build_task_context(task: Task, step: TaskStep) -> str:
    """构建注入到 MAEOS 的上下文。"""
    parts = []

    # 1. Task 目标
    parts.append(f"[TASK GOAL]\n{task.description}")

    # 2. 前序步骤输出（如果 step.order > 1）
    previous_steps = await task_state.get_completed_steps(task.id, before=step.order)
    for ps in previous_steps:
        parts.append(f"[PREVIOUS STEP {ps.order}: {ps.objective}]\n{ps.output}")

    # 3. Workspace memory（相关条目）
    relevant = await workspace_memory.search(task.workspace_id, task.description)
    if relevant:
        parts.append(f"[WORKSPACE MEMORY]\n{chr(10).join(relevant)}")

    # 4. Channel 最近对话摘要
    recent = await get_channel_summary(task.channel_id)
    if recent:
        parts.append(f"[CHANNEL RECENT]\n{recent}")

    return "\n\n---\n\n".join(parts)
```

### 7.3 生命周期

| 阶段 | 操作 |
|---|---|
| Task 创建 | 从 workspace/channel 采集初始 context |
| Step 执行前 | task_memory.build_context() 构建当前步骤的 input_context |
| Step 完成后 | 结果存入 TaskStep.output；写入 workspace_memory（可选） |
| Task 完成后 | 将 Task 摘要写入 workspace_memory（决策记录） |

## 8. 失败恢复设计

### 8.1 利用现有基础设施

| 现有能力 | 应用场景 |
|---|---|
| `RetryPolicy.classify()` | 对 TaskStep 失败分类（VALIDATION / LOGIC / SYSTEM） |
| `RetryPolicy.decide()` | 计算是否重试、延迟策略 |
| `BackoffStrategy` | 重试间隔（FIXED / LINEAR / EXPONENTIAL） |
| `TraceLogger` | 记录 Task 执行全链路 trace |
| `ExecutionMemory.replay()` | Debug：回放 Task 执行过程 |

### 8.2 Task 级恢复策略

#### 暂停（Pause）

```python
async def pause_task(task_id: str):
    """暂停 Task——当前正在执行的 step 完成后不再调度下一个 step。"""
    task = await task_state.get(task_id)
    if task.status != EXECUTING:
        raise InvalidStateError()
    task.status = PAUSED
    # task_executor._loop 会在 step 完成后检查此标志
```

#### 恢复（Resume）

```python
async def resume_task(task_id: str):
    """恢复已暂停的 Task——从暂停位置继续。"""
    task = await task_state.get(task_id)
    if task.status != PAUSED:
        raise InvalidStateError()
    task.status = EXECUTING
    # 重新启动 execute_loop，跳过已完成的 steps
```

#### 重试（Retry）

```python
async def retry_step(task_id: str, step_id: str):
    """重试一个失败的 step。复用现有 RetryPolicy。"""
    step = await task_state.get_step(step_id)
    # 重置 step 状态
    step.status = PENDING
    step.error = ""
    step.retry_count += 1
    # 重新调度
    await task_executor.execute_step(step)
```

#### 失败降级（Degrade）

当 step 多次重试仍失败时：

```
SYSTEM_FAIL ──► abort task（基础设施问题，重试无意义）
LOGIC_FAIL   ──► 切换 teammate / 换模型重试
                  如果仍失败 ──► skip step（标记 SKIPPED），继续后续 steps
VALIDATION_FAIL ──► 重新规划（re‑plan）当前 step
```

#### 人工确认（Human Approval）

```python
async def request_approval(step: TaskStep, reason: str):
    """发出人工确认请求，暂停执行直到确认。"""
    await emit_sse({
        "type": "task_approval_required",
        "task_id": step.task_id,
        "step_id": step.id,
        "reason": reason,
        "step_output": step.output,
    })
    # 等待 POST /api/tasks/{id}/approve 或 reject
```

## 10. 风险分析

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| **Planner hallucination**：LLM 将简单请求分解成过多步骤 | Task 执行慢、token 浪费 | Step 数量上限（默认 5），支持用户手动编辑 steps |
| **并发 Task 堵塞 MAEOS**：多个 Task 同时提交耗尽了 Worker Pool | Chat 响应延迟上升 | Task Layer 引入自己的并发限制（max_concurrent_tasks），优先保障 Chat |
| **Memory context 过大**：大量前序步骤输出撑爆 LLM context | 推理质量下降、成本升高 | Truncate 策略：只保留最近 N 个步骤的输出摘要 |
| **状态不一致**：Task 状态和 MAEOS 内部 Task 状态不同步 | 前端展示错误 | TaskExecutor 定期 sync；MAEOS 侧加查询接口 |
| **暂停后恢复失败**：长时间暂停导致 context 过时 | 恢复后输出质量下降 | 恢复时重新构建 context（基于当前 workspace_memory） |
| **无侵入约束被突破**：Team 成员在开发中直接修改了 MAEOS | 稳定性风险 | Code review 强制 gate；文档明确标注 "do not touch" 文件列表 |

## 11. 未解决问题（Open Questions）

1. **Task 执行超时**：一个 Task 可能包含 10+ steps，总耗时可能数分钟。是否需要全局超时？
2. **Step 并行化**：相互独立的步骤是否应该并行执行（并发提交到 MAEOS）？Phase B 先实现串行，Phase D 评估是否加并行。
3. **Task 持久化**：Task/Step/Execution 存入 PostgreSQL 还是沿用 SQLite？建议与现有 Message 表同库。
4. **前端交互**：Task 列表、步骤详情、暂停/恢复/重试按钮——这些是否放在当前 channel UI 中，还是独立面板？
5. **Task 模板**：是否需要预定义 Task 模板（如 "代码审查" = 固定步骤序列）？
6. **Intent Classification 位置**：在 Channel route 层还是在 Task API 层？建议在 Channel route 层，实现 chat/task 透明切换。
