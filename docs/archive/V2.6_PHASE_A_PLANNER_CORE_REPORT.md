## 3. 架构总览

```
User Goal
    │
    ▼
TaskPlannerDriver.generate_plan()
    │
    ├── 1. _build_planner_prompt() — 组装 PLANNER_SYSTEM_PROMPT + goal + context
    │
    ├── 2. MAEOS.submit(description=prompt, wait=True) ← 强制走 MAEOS，不直接调 LLM
    │     └── FSMWorker → run_pipeline → 返回 LLM text
    │
    ├── 3. parse_plan(result) ← 解析/校验 pipeline
    │     ├── _extract_json — 支持 raw / ```json``` / 嵌入文本
    │     ├── _validate_schema — 必填字段 + 类型 + risk_level + confidence range
    │     ├── _validate_order — 1-based 连续，无重复
    │     └── _validate_dependencies — 无缺失 ref、无自引用、无循环（DFS）
    │
    ├── 4. validate_plan(plan) — post-parse 警告（低 confidence / 多步骤 / 负成本）
    │
    └── 5. return TaskPlan
```

## 4. 关键设计决策

### 4.1 Schema 设计

**TaskPlan** 包含 `confidence` + `rationale`（替代架构文档中的 `reasoning`）：

```python
@dataclass
class TaskPlan:
    task_id: str
    title: str
    description: str
    steps: list[TaskStepProposal]
    confidence: float         # 0.0–1.0，整体规划置信度
    rationale: str            # 规划推理过程（替代旧的 reasoning）
    estimated_total_cost: float
    risk_level: str           # LOW / MEDIUM / HIGH
    created_at: float
```

**TaskStepProposal** 每步带独立 `confidence` + `rationale`：

```python
@dataclass
class TaskStepProposal:
    order: int
    teammate_id: str
    objective: str
    expected_output: str = ""
    input_context_hint: str = ""
    depends_on: list[int]
    risk_level: str           # LOW / MEDIUM / HIGH
    confidence: float         # 0.0–1.0
    rationale: str            # 该步骤的存在理由
    ...
```

### 4.2 MAEOS 集成策略

- **约束**: 必须经过 `MAEOS.submit()`，禁止直接 `_call_llm()`
- **妥协（Phase A 范围限制）**: 当前 `FSMWorker.execute()` 使用固定通用 system_prompt。Phase A 将
  `PLANNER_SYSTEM_PROMPT` **嵌入到 user_message 头部**，使通用 assistant 仍能输出规划 JSON。
- **后续改进空间**: Phase B/C 可修改 MAEOS 或 pipeline，支持按 intent 切换 system_prompt。

### 4.3 重试策略

```
MAX_PLAN_RETRIES = 2
PLANNER_TIMEOUT = 120s

attempt 1 → parse fail? → attempt 2 → still fail? → raise PlanningError
MAEOS task failed? → retry → still fail? → raise PlanningError
```

## 5. 错误处理体系

| 错误 | 触发条件 |
|------|----------|
| `PlannerJSONError` | LLM 输出不含可提取的 JSON |
| `PlannerSchemaError` | 缺少必填字段、类型错误、risk_level/confidence 越界 |
| `PlannerEmptyPlanError` | steps 数组为空 |
| `PlannerOrderError` | order 不连续、重复、不从 1 开始 |
| `PlannerDependencyError` | depends_on 引用不存在的 order、自引用、循环依赖 |
| `PlanningError` | 重试耗尽后仍失败 |

### 6.3 边界情况

- `confidence` 严格限制 `[0.0, 1.0]`（包含边界）
- `risk_level` 仅接受 `LOW / MEDIUM / HIGH`
- 循环依赖检测使用 DFS + `recursion_stack`，可正确处理链表/反向/复杂环
- Forward dependency（步骤 n 依赖未来的步骤 m）**允许但记 warning**——非错误

## 7. 约束验证

| 约束 | 实现 |
|------|------|
| ✅ 不接 Workspace Memory | Phase A 范围外，`TaskPlannerInput.context` 预留接口但 Driver 不自动注入 |
| ✅ 不接 Planner 自动执行 | `generate_plan()` 仅返回 `TaskPlan`，不触发任何 Step 执行 |
| ✅ 不修改 MAEOS | Driver 只调用 `maeos.submit()` + `maeos.wait()`，不修改 MAEOS 代码 |
| ✅ 不修改 TaskExecutor | 不涉及 |
| ✅ 不实现 DAG 执行 | `depends_on` 仅作为元数据保留（Phase B 使用） |
| ✅ 使用 MAEOS.submit() | 所有 LLM 交互经过 MAEOS Pipeline |
| ✅ 禁止直接调用 LLM | 无 _call_llm 调用 |
| ✅ Confidence + Rationale | 取代旧的 reasoning 字段 |
