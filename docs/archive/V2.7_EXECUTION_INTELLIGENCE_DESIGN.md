## 1. 设计目标

当前 Task 执行系统（v2.5–v2.6）对执行结果的反馈非常浅：

- Step 只有 `COMPLETED` / `FAILED` 二元判定
- 没有结构化的质量评估
- 失败原因仅靠 RetryPolicy 的关键词匹配
- 执行结果与 Planner 的预期之间没有对比机制
- Re-plan 只靠人工触发（FAILED → PLANNING 是唯一定义好的路径）

**V2.7 Execution Intelligence** 旨在建立**闭环的智能执行反馈系统**：

| 维度 | 现状 | 目标 |
|------|------|------|
| 执行结果 | 二元（COMPLETED/FAILED） | 结构化的结果元数据 + 质量评分 |
| 计划对比 | 无 | 执行输出 vs 计划预期对比 |
| 质量评估 | 无 | 多维评分（完整度、一致性、准确性） |
| 失败分类 | 关键词匹配（3 类） | 结构化分类法 + 上下文感知 |
| Re-plan 触发 | 仅手动 FAILED→PLANNING | 条件驱动的自动/半自动重规划 |

## 2. 现状分析

### 2.1 现有数据流

```
TaskPlan (Planner) → save_plan() → convert_plan_to_steps()
    → TaskExecutor._execute_single_step()
        → MAEOS.submit() → MAEOS.wait()
        → handle_step_success() / handle_step_failure()
```

### 2.2 已有组件

- **TaskResultHandler** — 记录 execution 记录（token、cost、duration、output snapshot）
- **TaskEventLogger** — 结构化生命周期事件
- **TraceLogger** — 可观测性 trace
- **RetryPolicy** — 失败分类 + 重试决策
- **TaskContextBuilder** — 为每个 step 构建执行上下文（含 prior steps）

### 2.3 关键缺口

1. **没有 `ExecutionResult` 的独立模型** — 只记录了原始输出文本和误差信息，没有结构化的结果分析
2. **没有质量评分** — 无法回答"这个 step 的执行质量如何"
3. **计划对比缺失** — TaskPlan 的 `validation_criteria`、`expected_output` 字段从未被使用
4. **失败分类太粗** — 仅 error 文本关键词匹配，没有结构化的上下文维度
5. **Re-plan 无自动策略** — 没有定义"什么条件下应该重规划"

## 3. 核心概念

```
ExecutionResult          — 单次 step 执行的完整反馈记录
  ├── outcome            — SUCCESS / FAILURE / PARTIAL / SKIPPED
  ├── quality_score      — 质量评分（0.0 – 1.0）
  ├── quality_dimensions — 多维评估
  ├── classification     — 失败分类（当 outcome ≠ SUCCESS）
  └── plan_deviation     — 与计划预期的偏差

PlanDeviation            — 执行输出与计划预期的对比结果
  ├── output_matches_expected?    — 布尔
  ├── validation_criteria_passed? — 布尔（如果有 criteria）
  ├── deviation_detail            — 描述偏差
  └── severity                    — NONE / MINOR / MAJOR / CRITICAL

QualityScore             — 多维质量评分
  ├── completeness       — （0.0–1.0）输出是否完整覆盖了 objective
  ├── coherence          — （0.0–1.0）输出逻辑是否自洽
  ├── accuracy           — （0.0–1.0）输出是否准确（需未来结合外部验证）
  └── overall            — 加权综合

ExecutionClassification  — 结构化失败分类
  ├── category           — 一级分类
  ├── subcategory        — 二级分类
  ├── trigger            — 触发条件
  └── is_recoverable     — 是否可能通过重试/重规划恢复
```

## 4. Step Execution Result 生命周期

### 4.1 状态机扩展

当前 Step 状态机：

```
PENDING → RUNNING → COMPLETED
                  → FAILED
FAILED  → PENDING (retry)
```

v2.7 引入 `ExecutionResult` 作为 step 执行的**独立反馈记录**，不改变 step 状态机，但**在 COMPLETED/FAILED 时附加反馈数据**：

```
Step COMPLETED
  └→ 创建 ExecutionResult (outcome=SUCCESS)
      └→ Quality Evaluation
      └→ Plan Comparison
      └→ 如果 plan_deviation.severity ≥ MAJOR → 可能触发 re-plan signal

Step FAILED
  └→ 创建 ExecutionResult (outcome=FAILURE)
      └→ Failure Classification
      └→ 如果 is_recoverable=true → retry
      └→ 如果 is_recoverable=false → re-plan signal
```

### 4.2 结果生命周期阶段

```
┌─────────────────────────────────────────────────────────────────────┐
│  ExecutionResult Lifecycle                                          │
│                                                                     │
│  CREATED ─→ EVALUATED ─→ COMPARED ─→ (CLOSED / REPLAN_TRIGGERED)   │
│                                                            │        │
│                    REPLAN_TRIGGERED ────────────────────────┘        │
└─────────────────────────────────────────────────────────────────────┘
```

- **CREATED** — 原始结果被记录（当前 `handle_step_success/failure` 之后）
- **EVALUATED** — 质量评估完成
- **COMPARED** — 与计划对比完成
- **CLOSED** — 结果闭环，无需进一步操作
- **REPLAN_TRIGGERED** — 结果触发了重规划

### 4.3 创建时机

在 `TaskExecutor._execute_single_step()` 中，当前在 MAEOS 返回后已经有一段处理逻辑。v2.7 将在此处**新增**：

```
MAEOS COMPLETED:
  1. handle_step_success()          ← 现有
  2. record_execution()             ← 现有
  3. create_execution_result()      ← 新增：创建 ExecutionResult
  4. evaluate_quality()             ← 新增：质量评估
  5. compare_with_plan()            ← 新增：计划对比
  6. check_replan_trigger()         ← 新增：检查是否需要重规划

MAEOS FAILED:
  1. handle_step_failure()          ← 现有
  2. record_execution()             ← 现有
  3. create_execution_result()      ← 新增
  4. classify_failure()             ← 新增：结构化分类
  5. decide_next_action()           ← 现有 RetryPolicy 决策增强
```

## 5. Execution 与 TaskPlan 对比

### 5.1 对比维度

当前 TaskStepProposal 有字段从未被利用：
- `expected_output` — 预期输出描述
- `validation_criteria` — 验证标准
- `input_context_hint` — 上下文提示

v2.7 需要建立执行输出与这些字段的对比机制。

### 5.2 对比方式

| 方法 | 描述 | 适用场景 |
|------|------|----------|
| **LLM 评判** | 将 output + expected_output + validation_criteria 提交给 LLM 判断 | 通用，默认方案 |
| **确定性匹配** | 正则/关键词匹配 expected_output 中的特定模式 | expected_output 有结构化要求 |
| **规则引擎** | 基于 validation_criteria 的规则判断 | 有明确规则时 |

**推荐主方案：LLM 评判**，避免引入新的外部服务。

### 5.3 对比结果

```python
@dataclass
class PlanDeviation:
    output_matches_expected: bool           # 输出是否符合预期
    validation_criteria_passed: bool        # 验证标准是否通过（默认 True 如果无 criteria）
    deviation_detail: str = ""              # 偏差描述
    severity: str = "NONE"                  # NONE / MINOR / MAJOR / CRITICAL

    # 对比时使用的上下文
    expected_output: str = ""
    validation_criteria: str = ""
```

- **NONE**: 输出完全符合预期
- **MINOR**: 输出与预期略有差异，但不影响整体目标
- **MAJOR**: 输出明显偏离预期，需要关注
- **CRITICAL**: 输出完全不符合预期，必须干预

### 5.4 谁执行对比

新增 `ExecutionEvaluator` 服务（见 10.2 节），通过轻量 LLM 调用完成对比。

```
ExecutionEvaluator.compare(output, expected_output, validation_criteria)
    → PlanDeviation
```

**约束**：
- 对比 LLM 调用必须是**独立的、低优先级**的 MAEOS 任务（不阻塞主执行流）
- 对比结果异步写入，不影响 step 的状态机

## 6. Quality Evaluation

### 6.1 评估维度

```python
@dataclass
class QualityScore:
    # 核心维度
    completeness: float = 0.0    # 0.0–1.0: 输出是否完整覆盖了 objective
    coherence: float = 0.0       # 0.0–1.0: 输出逻辑是否自洽
    accuracy: float = 0.0        # 0.0–1.0: 输出是否准确（需要外部验证时为 estimates）

    # 元数据
    overall: float = 0.0         # 加权综合
    evaluator: str = "llm"       # 评估方式（llm / rule / manual）
    confidence: float = 0.0      # 评估本身的置信度
    details: dict = field(default_factory=dict)  # 详细评分理由
```

### 6.2 加权公式

```
overall = completeness * W_c  +  coherence * W_h  +  accuracy * W_a

默认权重：W_c=0.4, W_h=0.3, W_a=0.3
可在 TaskPolicy 中配置
```

### 6.3 评估方式

同对比机制，使用 LLM 评判，但这是**独立的两步**：

1. Quality Evaluation → 产出 QualityScore
2. Plan Comparison → 产出 PlanDeviation

**为什么分开？** Quality 关注"输出本身好不好"，PlanDeviation 关注"输出是否达到了计划的预期"。前者是绝对值，后者是对计划的相对值。

### 6.4 特殊处理

- **未定义 expected_output** 时，completeness 评估基于 objective 的覆盖程度
- **FAILED 步骤**：quality_score 直接设为 0.0，不进行质量评估
- **SKIPPED 步骤**：quality_score 设为 null/None

## 7. Failure Classification

### 7.1 现状

当前 `RetryPolicy.classify()` 基于 error 文本关键词匹配：

```python
SYSTEM_FAIL: timeout, connection, network, dns, refused, reset, 502, 503, 504
LOGIC_FAIL: json parse, invalid format, schema, logic, reasoning
VALIDATION_FAIL: (via flag)
UNKNOWN: everything else
```

### 7.2 v2.7 结构化分类法

```python
@dataclass
class ExecutionClassification:
    category: str               # 一级分类
    subcategory: str = ""       # 二级分类
    trigger: str = ""           # 触发条件描述
    is_recoverable: bool = True  # 是否可通过重试/重规划恢复
    confidence: float = 0.0     # 分类置信度
    detail: str = ""            # 详细说明
```

**分类法**：

| 一级类别 | 二级类别 | is_recoverable | 示例 |
|----------|----------|----------------|------|
| `SYSTEM` | `TIMEOUT` | ✅ | MAEOS 超时 |
| `SYSTEM` | `NETWORK` | ✅ | 网络中断 |
| `SYSTEM` | `RATE_LIMITED` | ✅ | 频率限制 |
| `SYSTEM` | `PROVIDER_ERROR` | ❌ | Provider 返回 500 |
| `SYSTEM` | `AUTH_FAILED` | ❌ | API Key 失效 |
| `LOGIC` | `PARSE_ERROR` | ✅ | JSON 解析失败 |
| `LOGIC` | `SCHEMA_ERROR` | ✅ | 输出格式不符合 schema |
| `LOGIC` | `INCOMPLETE_OUTPUT` | ✅ | 输出被截断 |
| `LOGIC` | `HALLUCINATION` | ❌ | 明显错误的推理 |
| `LOGIC` | `CONTRADICTION` | ❌ | 输出自相矛盾 |
| `QUALITY` | `LOW_SCORE` | ✅ | 质量评分低于阈值 |
| `QUALITY` | `PLAN_DEVIATION` | ✅ | 执行偏离计划 |
| `POLICY` | `COST_LIMIT` | ❌ | 超出成本预算 |
| `POLICY` | `RISK_BLOCKED` | ❌ | 风险评估阻止执行 |
| `POLICY` | `APPROVAL_REQUIRED` | ✅ | 需要人工审批（暂停） |
| `UNKNOWN` | — | ❌ | 无法分类 |

### 7.3 分类策略

**增强 RetryPolicy，不替换它**。

1. 先用现有关键词匹配快路径分类（SYSYTEM / LOGIC / UNKNOWN）
2. 如果出错前有质量评分，且评分低但无关键词命中 → 归入 QUALITY
3. 如果 strategy 返回 ABORT，但原因不明 → 用 LLM 分析 error + context 进行深度分类
4. LLM 深度分类为**异步、低优先级**，不阻塞重试决策

### 7.4 重试决策增强

当前 `RetryPolicy.decide()` 只输出 RETRY / FALLBACK / ABORT。

v2.7 增强：

```python
@dataclass
class EnhancedRetryDecision:
    action: str              # RETRY / FALLBACK / ABORT / REPLAN
    reason: str
    delay_ms: int = 0
    failure_type: str = ""
    suggested_replan: bool = False          # 是否建议重规划
    suggested_replan_scope: str = "STEP"    # STEP / TASK / CONTEXT
```

新增 `REPLAN` action —— 不同于 ABORT（彻底放弃），REPLAN 意味着"当前 step 不可恢复，但整个 task 可以重新规划绕过它"。

## 8. Re-plan Trigger 条件

### 8.1 触发类型

| 编号 | 触发条件 | 严重度 | 行为 |
|------|----------|--------|------|
| T1 | 单步 FAILED，且不可恢复，且还有替代方案 | MAJOR | 标记 step 为 FAILED，触发 task 级重规划 |
| T2 | 质量评分 < 阈值（默认 0.3） | MAJOR | 标记 step 质量不合格，触发 step 级重规划 |
| T3 | PlanDeviation.severity ≥ MAJOR | MAJOR | 标记偏差，触发 step 或 task 级重规划 |
| T4 | 连续 N 步 QUALITY.LOW_SCORE | CRITICAL | 暂停 task，通知人工，或整体重规划 |
| T5 | 成本超出 plan 预估 2x | MAJOR | 标记为成本违规，触发 task 级重规划 |
| T6 | Manual trigger（API / UI） | 任意 | 用户主动触发重规划 |

### 8.2 重规划范围

| 范围 | 含义 | 适用场景 |
|------|------|----------|
| `STEP` | 仅替换当前 failed step | T2（低质量）、T3（轻度偏差） |
| `TASK` | 从当前点重新规划剩余 steps | T1、T3（重度偏差）、T5 |
| `CONTEXT` | 重新构建上下文后重规划整个 task | T4（系统性失败） |

### 8.3 触发流程

```
触发条件被满足
  └→ ExecutionResult.status = REPLAN_TRIGGERED
  └→ 根据 scope 决定重规划策略

  STEP scope:
    └→ 修改当前 step 的 objective/teammate 后重新调度执行
    └→ 不需要调用 Planner

  TASK scope:
    └→ TaskPlanService.get_plan() → 获取现有 ACTIVE plan
    └→ 从当前步骤开始，用 Planner 重新生成剩余 steps
    └→ 保留已完成的 steps 结果

  CONTEXT scope:
    └→ TaskPlanService 将当前 plan 标记为 SUPERSEDED
    └→ 构建新的 planner context（含已执行的 steps + 失败分析）
    └→ 调用 Planner 生成全新 plan
    └→ 新计划从当前完成点开始
```

### 8.4 防止无限循环

每个 trigger 都附带 `cooldown`：

```
- 同一步在同一 plan 中最多触发 1 次 replan
- 同一个 task 在 5 分钟内最多触发 3 次 replan
- 达上限后自动暂停 task (→ PAUSED)，等待人工
```

### 8.5 配置化

在 TaskPolicy 中扩展：

```python
# 新增字段（作为 TaskPolicyModel 的一部分或独立配置）
replan_config = {
    "enabled": True,
    "quality_threshold": 0.3,
    "deviation_threshold": "MAJOR",
    "max_replans_per_task": 3,
    "max_replans_per_step": 1,
    "cooldown_seconds": 300,
    "auto_replan": True,      # True=自动, False=仅通知
}
```

## 9. 数据模型建议

### 9.1 新增模型

```sql
-- ExecutionResult: 单步执行的结构化反馈
CREATE TABLE execution_results (
    id              TEXT PRIMARY KEY,
    task_step_id    TEXT NOT NULL REFERENCES task_steps(id) ON DELETE CASCADE,
    task_execution_id TEXT NOT NULL REFERENCES task_executions(id) ON DELETE CASCADE,

    outcome         TEXT NOT NULL DEFAULT 'SUCCESS',   -- SUCCESS / FAILURE / PARTIAL / SKIPPED

    -- Quality Score
    completeness    REAL DEFAULT 0.0,
    coherence       REAL DEFAULT 0.0,
    accuracy        REAL DEFAULT 0.0,
    overall_quality REAL DEFAULT 0.0,

    -- Plan Deviation
    plan_matched    TEXT DEFAULT 'NONE',   -- NONE / MINOR / MAJOR / CRITICAL
    plan_deviation_detail TEXT DEFAULT '',

    -- Failure Classification (when outcome = FAILURE)
    failure_category    TEXT DEFAULT '',    -- SYSTEM / LOGIC / QUALITY / POLICY / UNKNOWN
    failure_subcategory TEXT DEFAULT '',
    is_recoverable      TEXT DEFAULT '1',  -- '0' / '1'

    -- Evaluation metadata
    evaluator           TEXT DEFAULT 'llm',
    evaluation_confidence REAL DEFAULT 0.0,

    -- Lifecycle
    status          TEXT NOT NULL DEFAULT 'CREATED',  -- CREATED → EVALUATED → COMPARED → CLOSED / REPLAN_TRIGGERED
    replan_triggered TEXT DEFAULT '0',     -- '0' / '1'
    replan_scope    TEXT DEFAULT '',       -- '' / STEP / TASK / CONTEXT

    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP,

    -- Indexes
    INDEX idx_execution_results_step (task_step_id),
    INDEX idx_execution_results_status (status),
    INDEX idx_execution_results_outcome (outcome)
);
```

### 9.2 现有模型修改

**TaskPolicyModel** 新增字段：

```sql
ALTER TABLE task_policies ADD COLUMN replan_config TEXT DEFAULT '{
    "enabled": true,

    "deviation_threshold": "MAJOR",
    "max_replans_per_task": 3,
    "max_replans_per_step": 1,
    "cooldown_seconds": 300,
```

### 9.3 不修改的模型

- **TaskStepModel** — 状态机不变，`output` 和 `error` 字段保留
- **TaskPlanModel** — 结构不变，`steps_json` 仍存储 plan 数据
- **TaskExecutionModel** — 保持原始执行记录，`execution_results` 作为其补充

### 10.2 ExecutionEvaluator

```python
class ExecutionEvaluator:
    """
    评估 step 执行结果的质量和计划偏差。

    职责：
    - evaluate_quality(output, objective) → QualityScore
    - compare_with_plan(output, expected_output, validation_criteria) → PlanDeviation

    方式：通过 MAEOS 提交低优先级 LLM 评判任务。
    异步：不阻塞主执行流，结果异步写入 DB。
    """

    async def evaluate(
        self,
        db: AsyncSession,
        result: ExecutionResultModel,
        step: TaskStepModel,
        plan_step: Optional[dict],   # 来自 TaskPlanModel.steps_json
    ) -> ExecutionResultModel:
        """执行质量评估 + 计划对比，更新 ExecutionResult 状态"""

    async def _llm_judge(
        self,
        output: str,
        objective: str,
        expected_output: str = "",
        validation_criteria: str = "",
    ) -> tuple[QualityScore, PlanDeviation]:
        """通过 MAEOS 提交 LLM 评判任务"""
```

```python
class ExecutionClassifier:
```

### 10.4 ReplanService

```python
class ReplanService:
    """
    Re-plan 触发决策和执行。

    职责：
    - check_triggers(result) → (should_replan, scope)
    - execute_replan(task, scope) → 执行重规划
    - cooldown tracking
    """

    async def check_and_trigger(
        self,
        db: AsyncSession,
        result: ExecutionResultModel,
        step: TaskStepModel,
        task: TaskModel,
        policy: TaskPolicyModel,
    ) -> bool:
        """检查是否需要重规划，如果需要则执行"""

    async def _check_quality_trigger(
        self, result, policy
    ) -> tuple[bool, str]:  # (triggered, scope)

    async def _check_deviation_trigger(
        self, result, policy
    ) -> tuple[bool, str]:

    async def _check_consecutive_failure(
        self, db, task_id, policy
    ) -> tuple[bool, str]:

    async def _execute_replan(
        self, db, task, scope: str, reviewer: Optional[str]
    ) -> None:
        """根据 scope 执行不同级别的重规划"""
```

## 13. 边界与约束

### 13.1 不做的事情

- **❌ 不修改 MAEOS** — 所有 LLM 交互仍通过 MAEOS
- **❌ 不修改 TaskExecutor** 的核心循环逻辑（仅追加调用点）
- **❌ 不实现 DAG** — 步骤仍是顺序执行
- **❌ 不引入外部评估服务** — 所有评估通过 MAEOS LLM
- **❌ 不修改现有状态机** — TaskStatus / TaskStepStatus 保持不变

### 13.2 必须遵守的原则

1. **不能增加 Step 执行的延迟主路径** — 质量评估、偏差对比、深度分类都是异步的
2. **Evaluation 必须是低优先级** — 使用 MAEOS 低优先级任务，不跟主执行争资源
3. **静默失败** — 评估/分类失败不导致 step 失败，仅记录 warning
4. **可配置** — replan 行为通过 TaskPolicy 控制，默认 auto_replan=false
5. **可观测** — 所有评估结果都持久化，可通过 API 查询

### 13.3 实施阶段建议

| 阶段 | 内容 | 产出 |
|------|------|------|
| Phase A | ExecutionResult 模型 + 持久化 | 基础数据层 |
| Phase B | ExecutionEvaluator（质量评估 + 计划对比）| 评估能力 |
| Phase C | ExecutionClassifier（结构化失败分类）| 分类能力 |
| Phase D | ReplanService（触发器 + 重规划执行）| 闭环能力 |
| Phase E | API 暴露 + 前端展示 | 用户可见 |
