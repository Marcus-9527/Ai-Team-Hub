# V3.0 Phase B — 前后端融合改造报告

> **日期**: 2026-07-07
> **目标**: 将 Task/Execution 能力融入 Chat 流程，创建内嵌式 TaskWorkspace，可视化 ExecutionResult，实现 Approval UI，优化状态同步
>
> **核心约束**:
> - 不修改 MAEOS
> - 不修改 Task Runtime
> - 不修改 Planner
> - 保持 Chat 兼容（ChannelView 消息渲染流程不可破坏）

## 1. 现状分析 (Phase A 架构)

### 1.1 当前 SSE 事件流

```
SSE Stream (/api/messages/{channelId})
  │
  ├─ teammate_message     → ChannelView (聊天气泡)
  ├─ teammate_end         → 无操作
  ├─ task_started         ─┐   [ChannelView: 跳过渲染 → dispatchTaskEvent]
  ├─ step_started          ├→  taskEventBus.dispatch() → TaskContext (仅刷新列表)
  ├─ step_completed        │
  ├─ task_completed        │
  └─ approval_required    ─┘
                            → TaskListView / TaskDetailView (独立全页视图)
```

**问题**: Task 事件当前被静默消费（`console.log` + 转发总线），Chat 中不渲染任何可视化卡片。用户看不到任务进度、步骤执行状态、结果质量。

### 1.2 当前前后端组件关系

| 侧面 | 当前 |
|------|------|
| Chat (ChannelView) | 纯消息流，忽略 task 事件 |
| Task 入口 | 侧边栏按钮 → TaskListView → TaskDetailView (独立全页) |
| ExecutionResult | 仅后端存储，前端无展示 |
| Approval | 后端 pydantic 模型已定义，前端 TaskDetailView 仅展示静态列表 |
| 状态同步 | TaskContext 轮询 (10s) + taskEventBus (SSE 被动) |

### 1.3 后端事件发射源

- `backend/services/team_collaboration.py` — `emit_event()` 函数，SSE `data: JSON\n\n` 格式
- 事件类型通过 `type` 字段识别: `task_started`, `step_started`, `step_completed`, `task_completed`, `approval_required`
- 事件通过 `sendMessage` 的响应流（`AsyncGenerator[str, None]`）直接发送到前端

### 1.4 ExecutionResult 数据模型

`backend/models.py` 中 `ExecutionResultModel` 核心字段:

| 字段 | 类型 | 范围 | 说明 |
|------|------|------|------|
| `outcome` | String | SUCCESS/FAILURE/SKIPPED | 执行结果 |
| `completeness` | Float | 0.0–1.0 | 关键词覆盖率 / 长度代理 |
| `coherence` | Float | 0.0–1.0 | 句子、段落、连接词结构质量 |
| `accuracy` | Float | 0.0–1.0 (Phase B 保留 None) | 预留 LLM Judge |
| `overall_quality` | Float | 0.0–1.0 | `completeness×0.6 + coherence×0.4` |
| `evaluator` | String | "rule" | 评估策略标识 |
| `evaluation_confidence` | Float | 0.0–1.0 | 规则评估固定 1.0 |
| `plan_matched` | String | NONE/PARTIAL/FULL | 计划匹配度 |
| `plan_deviation_detail` | Text | | 偏差详情 |
| `failure_category` | String | SYSTEM/LOGIC/QUALITY/POLICY/UNKNOWN | 失败分类 |
| `is_recoverable` | String | "0"/"1" | 是否可恢复 |
| `replan_triggered` | String | "0"/"1" | 是否触发重规划 |
| `status` | String | CREATED/EVALUATED | 评估状态 |

`backend/services/task/evaluation/rule_evaluator.py` 中 `EvaluationResult`:

| 字段 | 计算方式 |
|------|----------|
| `completeness` | expected 关键词重叠 / objective 宽松重叠 / 长度代理 |
| `coherence` | 句子数 + 段落 + 平均句长 + 句末标点 + 连接词 |
| `overall_quality` | completeness × 0.6 + coherence × 0.4 |
| `accuracy` | None (预留) |

## 3. 任务 A — Chat 内嵌 Task Card

### 3.1 设计决策

**核心思路**: 在 ChannelView 的消息流中，task 事件不再被 `return` 跳过，而是渲染为专用 TaskCard 组件，与普通聊天气泡同列表显示。

**保持 Chat 兼容的关键**:
- TaskCard 使用独立的视觉标识（左侧色条/图标区分于聊天气泡）
- 不改变 `messages` 状态管理逻辑
- TaskCard 不参与 SSE 流式合并（`message_id` 去重不受影响）
- 每个 task 事件独立渲染为一个卡片消息

### 3.3 ChannelView 修改点

**当前代码** (`ChannelView.jsx:80-86`):

```javascript
if (isTaskEventType(type)) {
  dispatchTaskEvent(event);
  console.log('[ChannelView] task event:', type, event);
  return; // ← 跳过渲染
}
```

**改为**:

```javascript
if (isTaskEventType(type)) {
  dispatchTaskEvent(event);
  // 渲染为 TaskCard 气泡
  setMessages(prev => [...prev, {
    id: 'taskcard-' + message_id,
    role: 'task_card',          // 新角色，独立渲染
    task_event_type: type,
    task_event_data: event,
    created_at: new Date().toISOString(),
  }]);
  return; // 不继续 teammate_message 逻辑
}
```

**ChannelView 消息渲染增加**:

```jsx
if (msg.role === 'task_card') return <TaskCard key={msg.id} event={msg.task_event_data} />;
```

### 3.4 事件格式规范（后端 → 前端）

ChannelView 接收到的 task 事件需包含以下字段：

```json
{
  "message_id": "uuid",
  "type": "task_started|step_started|step_completed|task_completed|approval_required",
  "channel_id": "频道ID",
  "role": "",
  "phase": "",
  "payload": {
    "task_id": "任务ID",
    "task_title": "...",
    "step_id": "...",
    "step_order": 1,
    "total_steps": 5,
    "objective": "...",
    "teammate_id": "...",
    "status": "running|success|failure",
    "output": "...",
    "duration_ms": 2300,
    "quality": 0.78,
    "completeness": 0.85,
    "coherence": 0.72,
    "approval_id": "...",
    "approval_reason": "..."

  "timestamp": "2026-07-07T12:00:00Z"

```

> ⚠️ **注意**: 后端 `emit_event()` 当前 `payload` 内容较简略。Phase B 需要后端在 task 事件发送时补充 `task_title`、`total_steps`、`output`、`duration_ms` 等字段到 `payload` 中。建议在 `team_collaboration.py` 的 task 事件发射点从 `TaskExecutor` 或 `TaskState` 获取完整上下文后填充 payload。

## 7. 任务 E — 优化状态同步: SSE 优先 / Polling 后备

### 7.1 当前同步策略

| 机制 | 方式 | 间隔 | 问题 |
|------|------|------|------|
| taskEventBus | SSE (通过 `sendMessage` 响应流) | 实时 | 只在发送消息时触发，非持久连接 |
| TaskContext polling | REST GET | 10s | 延迟 + 带宽浪费 |

### 7.2 目标架构

```
永久 SSE 连接 (独立于 sendMessage)
  /api/tasks/{taskId}/events?channel_id=xxx
        │
    ┌───┴───┐
    │ 优先   │  连接成功 → 实时接收 task events
    │        │  连接断开 (5s 超时) → 降级为 polling
    │ 后备   │  polling 间隔: 3s (连续 3 次成功 → 重试 SSE)
    └───────┘
```

### 7.3 taskSSE.js 服务

```javascript
// services/taskSSE.js
export class TaskSSEService {
  constructor(taskId, onEvent, onStatusChange) {
    this.taskId = taskId;
    this.onEvent = onEvent;
    this.onStatusChange = onStatusChange;
    this._pollTimer = null;
    this._abortController = null;
    this._mode = 'sse'; // 'sse' | 'polling'
  }

  connect() { /* EventSource or fetch-based SSE */ }
  _startPolling() { /* 3s interval + backoff */ }
  disconnect() { /* cleanup */ }
}
```

### 7.4 后端 SSE 端点

```python
# routes/tasks.py 新增
@router.get("/{task_id}/events")
async def stream_task_events(
    task_id: str,
    request: Request,
    db: AsyncSession = Depends(get_db),
):
    """SSE 端点: 实时推送 Task 生命周期事件"""
    async def event_generator():
        while True:
            if await request.is_disconnected():
                break
            # 从 EventBus 或 task state 拉取增量事件
            events = await get_pending_events(task_id, db)
            for event in events:
                yield emit_event(event.type, ...)
            await asyncio.sleep(1)
    return StreamingResponse(event_generator(), media_type="text/event-stream")
```

### 7.5 Polling 后备逻辑

| 场景 | 行为 |
|------|------|
| SSE 连接成功 | 实时接收，不轮询 |
| SSE 断开 (5s 无数据) | 切到 polling (3s)，显示"实时连接中断"指示器 |
| Polling 中 SSE 重连成功 | 切回 SSE，关闭 polling |
| 页面不可见 (visibilitychange) | 暂停 polling，降低资源占用 |
| 连续 3 次 polling 错误 | 指数退避 (3s → 6s → 12s → max 30s) |
