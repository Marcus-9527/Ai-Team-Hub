# AI Team Hub v2.6 Planner Layer Architecture

> **阶段**: 架构设计  
> **日期**: 2026-07-07  
> **目标**: 设计 Task 自动规划能力，使 AI Team Hub 能根据用户目标自动分解为可执行 TaskStep  
> **约束**:
> - Planner 必须作为 MAEOS 管理的特殊 Teammate
> - ❌ 不创建新的 Planner Runtime
> - ❌ 不绕过 TaskExecutor
> - ❌ 不直接调用 LLM
> - ✅ 所有 Agent 执行必须经过 MAEOS Runtime
>
> **前置依赖**: v2.5 Task Execution Layer（TaskManager / TaskExecutor / TaskState / TaskPolicy / TaskContext）

---

## 1. 现状与问题

### 1.1 当前规划能力

v2.5 Task 层现有的规划能力只存在于 `pipeline.py` 的 `planner_step`：

```
pipeline.planner_step()
  └─ 简单的意图分类 + 查询重写
  └─ 输出: {"intent": "question", "rewritten_query": "..."}
  └─ 不产生结构化的 TaskStep
  └─ 不指定执行 teammate
  └─ 不评估风险/依赖
```

TaskExecutor 执行时需要 caller 已经准备好 `TaskStep[]`——目前没有自动从 User Goal 到 TaskStep 的转换机制。

### 1.2 缺失的能力

| 能力 | 现状 | 目标 |
|------|------|------|
| User Goal → TaskStep 分解 | 无 | Planner Teammate 输出 TaskPlan |
| 步骤依赖关系 | 无 | TaskStepProposal 含 dependency |
| 执行 teammate 推荐 | 无 | Planner 根据步骤性质推荐 |
| 风险等级评估 | 无 | Planner 输出 risk_level |
| 上下文感知规划 | 无 | Planner 读取 Workspace Memory + Channel Context |
| 规划失败恢复 | 无 | Policy 评估 + re-plan 流程 |

## 2. Planner Flow

### 2.1 完整执行流程

```
User Goal
    │
    ▼
[1] TaskManager.create_task(description=user_goal)
    │  status = CREATED
    │
    ▼
[2] TaskManager.start_planning(task)
    │  status = PLANNING
    │
    ▼
[3] Planner Teammate 激活
    │  MAEOS.submit(description=..., teammate_id="planner")
    │   ├─ 读取 Global Memory（全局规则/偏好）
    │   ├─ 读取 Workspace Memory（历史决策/上下文）
    │   └─ 读取 Channel Context（当前频道消息）
    │
    ▼
[4] Planner 输出 TaskPlan（TaskStepProposal[]）
    │
    ▼
[5] Policy Evaluation（复用 task_policy.py）
    │   ├─ risk_level 校验
    │   ├─ teammate 权限校验
    │   └─ 成本预估校验
    │
    ▼
[6] Approval（可选，由 Policy 决定）
    │   ├─ HIGH risk → 需要人工审批
    │   ├─ MEDIUM risk + approval_required → 需要人工审批
    │   └─ LOW risk → 自动放行
    │
    ▼
[7] TaskManager.save_plan(steps)
    │  status = EXECUTING
    │
    ▼
[8] TaskExecutor.execute_task(task)
      └─ for each step → MAEOS.submit() → collect result
```

### 2.2 Planner 激活方式

Planner 是一个**特殊的系统 Teammate**，注册在 MAEOS 管理的 Teammate Registry 中：

```python
# backend/services/team_registry.py 追加
TEAMMATE_DEFINITIONS["planner"] = {
    "role": TeammateRole.COORDINATOR,
    "system_prompt": "[IDENTITY LOCK]\nYou are Planner Teammate.\n"
                     "Your only job is to decompose user goals into a structured TaskPlan.\n"
                     "You do NOT execute tasks yourself.",
    "tools": ["plan", "decompose"],
}
```

激活点：`TaskManager.start_planning()` 内部构造一个 Planner 输入，提交到 MAEOS：

```python
# 概念代码（非实现）
async def start_planning(self, db, task):
    task = await self.transition_task_status(db, task.id, TaskStatus.PLANNING)

    # 构造 Planner 任务描述
    planner_input = TaskPlannerInput(
        goal=task.description,
        workspace_memory=workspace_mem.get_all(),
        channel_context=channel_messages[-50:],
        global_memory=global_rules,
        previous_tasks=task_history[-10:],
    )

    # 通过 MAEOS 提交（Planner 作为 teammate）
    maeos_task_id = await maeos.submit(
        description=planner_input.serialize(),
        intent=f"planning:{task.id}",
        wait=False,
    )
    # ... 等待完成 ...
    task_plan = parse_planner_output(maeos_task.result)
    return task_plan
```

## 3. Planner Output Schema

### 3.1 TaskPlan

```python
@dataclass
class TaskPlan:
    """Planner 的完整输出——包含规划元信息 + 所有步骤。"""
    task_id: str                    # 关联的 Task.id
    title: str                      # 规划标题（可不同于 task.title）
    description: str                # 规划说明
    steps: list[TaskStepProposal]   # 有序的步骤提案
    reasoning: str                  # 规划推理过程
    estimated_total_cost: float     # 预估总成本 (µ$)
    risk_level: str                 # 整体风险等级: LOW / MEDIUM / HIGH
    created_at: float               # 规划时间戳
```

### 3.2 TaskStepProposal

```python
@dataclass
class TaskStepProposal:
    """Planner 对单个步骤的提案。"""
    order: int                      # 执行顺序（1-based）
    teammate_id: str                # 推荐执行此步骤的 teammate（可为空，由 executor 按 policy 最终分配）
    objective: str                  # 步骤目标（即 step 的指令/任务描述）
    expected_output: str            # 期望输出描述（给 Teammate 的 guidance）
    input_context_hint: str         # 需要注入的上下文提示

    # 依赖关系
    depends_on: list[int]           # 依赖的上一步 order 列表（空 = 无依赖）

    # 风险评估
    risk_level: str                 # 此步骤的风险等级: LOW / MEDIUM / HIGH
    estimated_cost: float           # 预估成本 (µ$)
    estimated_tokens: int           # 预估 token 消耗

    # 质量门禁
    requires_approval: bool         # 是否需人工审批
    validation_criteria: str        # 验收标准描述
```

### 3.3 Planner 输出序列化

Planner 的输出直接是 JSON 化的 `TaskPlan`。由 `PlannerOutputParser` 解析后，通过 `TaskStateManager.create_step()` 写入 DB：

```python
# 解析后的映射关系
TaskStepProposal.order          → TaskStepModel.order
TaskStepProposal.objective      → TaskStepModel.objective
TaskStepProposal.teammate_id    → TaskStepModel.teammate_id
TaskStepProposal.requires_approval → TaskStepModel.requires_approval
TaskStepProposal.risk_level     → TaskPolicyModel.risk_level（Task 级别）
```

### 3.4 依赖解析映射

`depends_on` 字段记录的是 `order` 引用，语义上是顺序约束：

```text
steps: [
  { order: 1, objective: "分析需求",       depends_on: [] },
  { order: 2, objective: "设计架构",       depends_on: [1] },
  { order: 3, objective: "实现代码",       depends_on: [1, 2] },
  { order: 4, objective: "编写测试",       depends_on: [3] },
  { order: 5, objective: "Code Review",   depends_on: [3, 4] },
]
```

v2.6 仅支持**线性串行执行**（Phase B 要求），`depends_on` 作为元数据保留给未来的并行调度器。

## 4. MAEOS Integration

### 4.1 Planner 如何调用 MAEOS

Planner 本身是一个 Teammate，它的执行**必须经过 MAEOS Runtime**。流程图：

```text
TaskManager.start_planning(task)
    │
    ▼
TaskPlannerDriver（新增）
    │  不是新的 Runtime，是 TaskManager 内部的一个 Orchestrator Helper
    │
    ├── 1. 构建 Planner Input（含 context/memory）
    │
    ├── 2. MAEOS.submit(
    │        description = planner_input.to_prompt(),
    │        priority = TaskPriority.HIGH,
    │        intent = f"planner:{task.id}",
    │        wait = True,        # 同步等待规划完成
    │    )
    │     │
    │     ├──► PriorityTaskQueue ──► FSMWorker.execute()
    │     │      run_pipeline(
    │     │        system_prompt = PLANNER_SYSTEM_PROMPT,  # 新的规划专用 prompt
    │     │        user_message = planner_input_json,
    │     │      )
    │     │      └─► 返回 TaskPlan JSON
    │     │
    │     ◄── maeos_task.result = TaskPlan JSON
    │
    ├── 3. TaskPlanParser.parse(maeos_task.result)
    │     ├── 校验 JSON Schema
    │     ├── 校验 step 完整性（order 连续性、依赖可达性）
    │     └── return TaskPlan
    │
    ├── 4. TaskPolicyService.evaluate_plan(task, plan)
    │     ├── 逐 step 检查 policy
    │     └── 标记需要审批的步骤
    │
    └── 5. TaskStateManager.bulk_create_steps(task.id, plan.steps)
          task.status = EXECUTING
```

### 4.2 关键设计决策

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| Planner 调用方式是同步还是异步？ | submit+wait / submit 后轮询 | **submit+wait** | 规划阶段通常短（<30s），同步等待简化逻辑；Task 级异步由 TaskManager 保证 |
| Planner 使用现有 pipeline 还是新 pipeline？ | 复用 pipeline / 新路径 | **复用 run_pipeline** | 不创建新 Runtime；通过切换 system_prompt 切换行为 |
| Planner 的 LLM 调用路径 | 直接调用 ai_service / 通过 MAEOS | **通过 MAEOS** | 约束要求：不能绕过 MAEOS |

### 4.3 Planner 与现有 pipeline.py 的关系

不修改 `pipeline.py`。Planner 使用相同的 `run_pipeline()` 入口，但：

- `system_prompt` 替换为 `PLANNER_SYSTEM_PROMPT`（规划专用，非通用 assistant prompt）
- `max_tokens` 设为 4096（规划输出比单条回复长）
- `user_message` 是结构化的 `TaskPlannerInput` JSON

```python
# 概念：注入新的规划 prompt
PLANNER_SYSTEM_PROMPT = """\
You are the Planner Teammate in a multi-agent team.
Your ONLY responsibility is to decompose user goals into structured execution plans.

You MUST output a valid JSON TaskPlan object.
You do NOT write code, do NOT answer questions, do NOT execute tasks.
"""
```

## 5. Memory Integration

### 5.1 Planner 可访问的三层记忆

| 记忆层 | 来源 |  Planner 读取的时机 | 用途 |
|--------|------|---------------------|------|
| **Global Memory** | 系统级持久化配置（全局规则、偏好、约束） | 每次 planning 前 | 确保规划符合全局策略（如"所有代码必须含测试"、"优先使用 POSTGRES"） |
| **Workspace Memory** | `WorkspaceMemory`（决策/修订/推理轨迹） | 每次 planning 前 | 了解工作区上下文：哪些决策已做、哪些方案已淘汰 |
| **Channel Context** | `Channel.messages`（最近的对话历史） | 每次 planning 前 | 理解当前对话状态：用户情绪、讨论焦点、已确认的事项 |

### 5.2 记忆注入流程

```
TaskPlannerDriver.prepare_input(task, db)
    │
    ├── Global Memory
    │     └── backend/services/kernel/memory_kernel.py (stub → 实现)
    │         └── 读取全局规则列表（数据库或配置文件）
    │
    ├── Workspace Memory
    │     └── WorkspaceMemory.get_all() + get_decisions() + get_revisions()
    │         └── 序列化为 JSON context
    │
    └── Channel Context
          └── SELECT * FROM messages WHERE channel_id = task.channel_id
              ORDER BY created_at DESC LIMIT 100
              └── 序列化为消息摘要
```

### 5.3 TaskPlannerInput 格式

Planner 接收的结构化输入：

```json
{
  "goal": "为用户系统实现 JWT 认证",
  "context": {
    "global_rules": [
      "所有代码必须包含单元测试",
      "优先使用 async/await 模式"
    ],
    "workspace_decisions": [
      {"decision": "使用 FastAPI 框架", "reasoning": "团队熟悉度最高"},
      {"decision": "数据库使用 PostgreSQL", "reasoning": "数据一致性要求高"}
    ],
    "workspace_revisions": [
      {"original": "Flask", "revised": "FastAPI", "reason": "性能需求提升"}
    ],
    "channel_history": [
      {"role": "user", "content": "需要加用户认证系统，JWT 方式"},
      {"role": "engineer", "content": "建议用 fastapi-jwt-auth 包"}
    ]
  }
}
```

### 5.4 记忆持久化（规划后）

Planner 的输出（含推理过程）会写入 Workspace Memory 作为 Decision 记录，供后续 Task 引用：

```python
await workspace_memory.record_decision(
    thread_id=task.channel_id,
    decision=f"Task Plan: {task.title}",
    reasoning=plan.reasoning,
    actor=f"teammate:planner",
    metadata={"task_id": task.id, "steps_count": len(plan.steps)},
)
```

## 6. Failure Handling

### 6.1 规划失败场景

| 场景 | 检测方式 | 处理策略 |
|------|----------|----------|
| **MAEOS 执行超时** | `maeos.wait()` 抛出 TimeoutError | Task → FAILED，允许用户重新触发 planning |
| **Planner 输出非 JSON** | `TaskPlanParser` 解析失败 | 重试 1 次；仍失败 → Task → FAILED |
| **Planner 输出空步骤** | `len(steps) == 0` | Task → FAILED，reason="Planner returned empty plan" |
| **Planner 输出步骤顺序错误** | order 不连续或有循环依赖 | Task → FAILED，reason="Invalid step dependency" |
| **Policy 阻止所有步骤** | `TaskPolicyService.evaluate_plan` 全部 blocked | Task → PAUSED，请求人工干预调整 Policy |
| **Policy 要求审批** | `requires_approval == true` | Task → PAUSED，发出 Approval 事件 |

### 6.2 重试策略

```python
# 概念
MAX_PLAN_RETRIES = 2

async def plan_task(task, db):
    for attempt in range(1, MAX_PLAN_RETRIES + 1):
        try:
            plan = await _do_planning(task)
            if validate_plan(plan):
                return plan
        except (PlannerOutputError, TimeoutError) as e:
            logger.warning(f"Planning attempt {attempt} failed: {e}")
            if attempt == MAX_PLAN_RETRIES:
                raise PlanningFailedError(f"Planning failed after {MAX_PLAN_RETRIES} attempts")

    raise PlanningFailedError("Planning exhausted retries")
```

### 6.3 人工介入路径

```
Planning Failed
    │
    ├── Task.status = FAILED
    │     │
    │     ├── 用户点击"重新规划" → start_planning() 重新触发
    │     │
    │     └── 用户手动创建/编辑步骤 → TaskManager.create_step() → start_execution()
    │
    └── Task.status = PAUSED（Policy 阻止）
          │
          ├── 人工审批 → TaskPolicyService.update(risk_level=LOW) → resume()
          └── 人工驳回 → Task.status = CANCELLED
```

## 9. 后续迭代（v2.7+ 考虑）

| 能力 | 说明 |
|------|------|
| 并行步骤执行 | 利用 `depends_on` 解析 DAG，并行执行无依赖的步骤 |
| 自适应 re-plan | 执行中某步骤失败时，自动触发局部 re-plan |
| Planner 自我进化 | 根据历史 Task 完成情况，优化规划策略 |
| Multi-shot planning | Planner 输出多个候选方案 → 人工选择最优 |
| Context 压缩 | 长对话时自动摘要，避免 token 溢出 |
